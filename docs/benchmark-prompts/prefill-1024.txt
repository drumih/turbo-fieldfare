This prefill calibration prompt describes an engineering investigation into a single-prompt inference engine. The runtime should read the prompt in order, preserve causal context, append key and value rows, route experts deterministically, and compute logits only for the final prompt token. The passage is intentionally plain text so tokenizer counts can be reported by the benchmark before any model load. It asks for a concise answer about why chunked prefill reduces time to first token while keeping memory bounded.

The model is running on a small Apple Silicon system with an eight gigabyte physical memory ceiling. The implementation must never load a whole checkpoint, a whole shard, or a whole routed expert pool into Swift heap memory. Instead it should use memory mapped resident weights, tile bounded scratch buffers, explicit key value cache ranges, and streamed expert reads. The benchmark should make these constraints visible by printing prompt tokens, prefill wall time, skipped head count, final head time, GPU command buffer buckets, routed expert I/O, advisory read counters, resident set size, and physical footprint.

The prompt is also designed to exercise normal language patterns rather than a repeated single token. It includes nouns, verbs, punctuation, numbers, and technical phrases that resemble real requests. A correct response should explain that prefill is different from decode because many known prompt tokens can be processed as a bounded block, but each generated token still depends on the previous sampled token. It should mention that streaming experts remain necessary, batching is not required for this project, and the chunk size must be chosen by measurement rather than guesswork.

For the first implementation slice, the fast path should stay opt in. A fallback prefill mode may replay tokens with the existing scalar forward pass, but it should skip the expensive final head for every prompt token except the last one. That fallback is useful because it proves the public API, cancellation behavior, cursor advancement, progress reporting, fused greedy output mode, and sampling logits mode before the chunked kernels are ready. It is not the final performance target, and benchmark output must not pretend that fallback replay is chunked prefill.

For the second implementation slice, the key value cache needs explicit range views. A chunk writes several K and V rows before the global cursor advances, so the cache manager should expose temporary views with a caller supplied valid token count. The attention kernel then masks each query row independently: token zero sees only earlier context plus itself, token one sees one more row, and no query may read future rows in the same chunk. Sliding window layers clamp the first visible key by the window size, while full layers see the whole prefix.

For the next implementation slices, the projection kernels should move from repeated vector operations toward block shaped matrix work. The first custom kernel can be simple and correctness focused, but it must preserve the MLX affine four bit format with per group BF16 scale and BF16 bias. Router work should batch token rows, select top experts, and group token expert pairs by expert only after a route diversity benchmark shows the live expert tile size. The grouped expert path must not raise the default expert cache slot count.

A careful implementation must keep streaming semantics visible. The user interface may receive fewer progress events than before, because a chunk can complete several prompt tokens at once, but progress still has to be monotonic and must end at the prompt total. If a producer reports slice local progress, the caller must add the slice base before showing prompt global progress. If cancellation or an error happens after partial key value writes, the runner state is not reusable until reset clears the cache cursor and drops stale pages.

The attention path needs separate correctness and performance stages. The correctness stage can use a simple two pass softmax per output element. It should not allocate a score matrix, it should accept element strides, and it should test nonzero Metal buffer offsets. The performance stage can later tile keys, share row maxima across lanes, and reduce repeated dot products. The important invariant is that each query row has an absolute position equal to the chunk start plus the row index, and Gemma attention scale remains one point zero.

The full attention layers have a special key value detail. The raw key projection can be reused as the value projection input, but the cached K and V rows diverge before attention. K receives its per head norm and rotary position transform, while V receives its no scale norm and no rotary transform. A chunked path must preserve this quirk. A test should compare the chunked K and V rows against the scalar runner fixture before any real decode benchmark is allowed to promote the path.

The routed mixture of experts is the largest scheduling risk. A token has several selected routed experts and a dense shared expert path, so direct writes from every routed pair into the same output row would race. The deterministic first design writes unweighted routed outputs into a token rank partial buffer, then reduces token major with route weights. That costs bounded scratch but makes correctness easy to audit. Only after route diversity data is recorded should the implementation stream grouped expert tiles and tune the tile size.
